# implement — バグ改善・機能追加のワークフロー

バグ改善・機能追加を行うときは、以下の 5 ステップを **必ずこの順序で** 実行する。
ステップの省略・順序の入れ替えは禁止。

## 1. 最新の main から新しくブランチを切る

作業を始める前に、必ず最新の `main` を取得してからブランチを作成する。
既存の作業ブランチや古い `main` から派生させてはならない。

```bash
git fetch origin main
git switch -c <branch-name> origin/main
```

- ブランチ名は変更内容に応じたプレフィックスを付ける（例: `feat/...`, `fix/...`, `chore/...`）。
- 作業ツリーに未コミットの変更が残っている場合は、先にユーザーへ確認してから進める。

## 2. 実装後、テストを必ず実行する

実装が一区切りしたら、コミット前に必ずユニットテストを実行する。

```bash
npm test
```

- 失敗したテストがある場合は、修正して全件パスするまで次のステップへ進まない。
- 型エラーの検出には `npm run typecheck` も併用できる（`npm run build` にも含まれる）。

## 3. E2E テストも実行する

ユニットテストがパスしたら、E2E テストも必ず実行する。

```bash
npm run e2e
```

- E2E が失敗した場合も同様に、原因を修正して全件パスするまで先へ進まない。
- E2E 実行時のスクリーンショットは必ず `screenshots/` ディレクトリに出力する
  （`npm run e2e` のデフォルト出力先。`--out` で変更しないこと）。
- テスト結果（パス件数・失敗内容）はユーザーに報告する。

## 4. ユーザーに動作確認・承認を求める

テストがすべてパスしたら、**PR を作成する前に必ず** ユーザーへ動作確認と承認を求める。

- 変更内容の要約、実行したテストとその結果、動作確認の手順（拡張の読み込み方や確認対象のページなど）を提示する。
- `screenshots/` に出力された E2E スクリーンショットのうち、変更に関係するものを確認材料として提示する。
- AskUserQuestion などでユーザーに承認するかどうかを明示的に確認する。
- 承認が得られるまで、次のステップ（PR 作成）には進まない。修正依頼があれば対応し、ステップ 2〜4 をやり直す。

## 5. 承認後、main へ向けて PR を作成する

ユーザーの承認が降りたら、変更をコミット・プッシュし、`main` ブランチをベースにした PR を作成する。

```bash
git push -u origin <branch-name>
gh pr create --base main --title "<変更内容の要約>" --body "<変更の背景・内容・テスト結果>"
```

- PR 本文には、変更の目的・主な変更点・実行したテスト（ユニット / E2E）の結果を記載する。
- PR の URL をユーザーに報告して完了とする。
